2.2 R02 — LLM API Availability / Downtime
Field Value
ID R02
Source PRD (01), LLM Strategy (06), Architecture (02)
Probability High (4)
Impact High (4)
Score 16 — P0
Owner CTO
Real-World Context: In early 2026, concurrent outages in major inference APIs (powering Cursor,
Copilot, and other tools) caused widespread developer paralysis. Unlike local IDE outages, agentic
outages prevent automated PR generation, codebase indexing, and complex refactoring — effectively
halting DevOps pipelines.
Detailed Mitigation:
1. 4-provider failover chain — Primary →  Fallback 1 →  Fallback 2 →  Fallback 3 →  Cache →  Queue
2. Circuit breaker pattern — Auto-open after 3 consecutive failures; half-open after 30s timeout
3. Max 2s failover time — Health checks every 10s; mark unhealthy after 3 consecutive failures
4. Cache last successful responses — Redis-backed; serve cached on all-provider failure
5. Queue for retry — NATS JetStream dead letter queue for tasks that fail all providers
6. Graceful degradation — If no providers available, queue task and notify user; never fail silently
7. Provider health dashboard — Real-time status of all providers in Grafana
Monitoring:
Provider error rate per hour (alert > 5%)

4.4 R17 — pgvector Performance at Scale
Field Value
ID R17
Source Schema (03), Database Strategy (10)
Probability Medium (3)
Impact High (4)
Score 12 — P1
Owner CTO
Real-World Context:
The "10 Million Threshold": pgvector is highly performant under 10M vectors. Beyond that,
dedicated vector databases (Pinecone, Weaviate) become preferable.

--- PAGE 13 ---

pgvectorscale achieves ~470 QPS at 99% recall on 50M vectors, but is NOT available on AWS RDS.
HNSW indexes are memory-hungry — Building on millions of rows can consume tens of GB of
RAM, starving the database.
Write contention — Heavy concurrent inserts with large HNSW indexes cause lock contention.
"Effective" filtering — Earlier versions had "overfiltering" issues; pgvector 0.8.0+ introduced
Iterative Scans to fix this.
Detailed Mitigation:
1. HNSW indexes — Not IVFFlat; better for production workloads
2. Quantization — Use halfvec  (16-bit) or binary quantization to reduce storage 16-32x
3. Benchmark with 1M+ vectors — Before launch; measure p95 query latency
4. Monitor index build memory — Set maintenance_work_mem  carefully; too high = OOM, too
slow = long builds
5. REINDEX CONCURRENTLY  — Avoid downtime during index rebuilds
6. ANALYZE  regularly — Prevent query plan drift
7. Partition high-volume tables — Time-based partitioning for tasks, auditlog, usagerecords
8. Plan for pgvectorscale — If using RDS, plan migration to self-managed Postgres when exceeding
10M vectors
9. Hybrid search — Build Reciprocal Rank Fusion in application layer; pgvector doesn't support this
natively
Monitoring:
pgvector query latency p95 (target: < 100ms; alert: > 500ms)
Index size vs. available RAM
Write contention (locks held > 100ms)
